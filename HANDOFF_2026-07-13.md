# Lumveil 引き継ぎメモ (2026-07-13, Fable5/Sonnet5 → Codex)

対象: `D:\tool\Lumveil`(GitHub: https://github.com/dhinsk65-create/Lumveil、Public、現在HEAD: `ed3d623` v1.7)

## 今回のセッションでやったこと

暗所自動画質調整(AUTO)を全面刷新した。検討書 `dark_auto_quality_review_2026-07-12.md`(別AI作成)の「案B」をベースに、Fable5が設計・レビュー、Sonnet5サブエージェントが実装、実映像でのユーザーフィードバックを受けて反復修正した。

### 最終的な構成

- **主処理**: `shaders/lumveil_shadow_lift.glsl` によるシャドウリフト(暗部限定トーンカーブ、輝度0.5以上は無効化、黒は持ち上げない)
- **副処理**: コントラスト・彩度の軽い補助(値は縮小済み、後述)
- **モード**: 操作バーの `⚡ AUTO` ボタンをクリックするとポップアップメニュー(OFF/控えめ/標準/見やすさ優先/極暗)。ボタン自体はテキスト固定「⚡ AUTO」で、文字色がモードを示す(`RT_MODE_COLORS`)
- **極暗モード**: 黒潰れがひどい動画向けの特別モード。ユーザーが手動レシピ(mpv輝度+33/コントラスト+89)で黒潰れ動画が見えることを実証したので、それを`dark_factor`連動で再現するよう実装(`_rt_mode == "極暗"`のときのみ有効)

### 重要な設計判断とその理由(やり直さないための記録)

1. **極暗シーンの自動検出は断念した**。「平均輝度が低い×暗部率が高い」「+黒潰れ率」の2段階で自動判定を試みたが、動画ごとに指標が重なり合い「対象場面で発動しない」「別動画で誤発動して白浮きする」の両方が解決できなかった(2ストライクで撤退)。**軽量な統計指標だけでの局所異常検出は、以前の「局所白飛び自動検出」(`project-lumveil-auto-adjust.md`参照)でも同じ理由で断念しており、この種の判定は本プロジェクトで繰り返し壁にぶつかっている構造的な限界と考えてよい**。今後もし再挑戦するなら、機械学習ベース以外の軽量手法は再検討しても時間の無駄になる可能性が高い。
2. **AUTOの発振バグ**: `player.screenshot_raw()` は mpv equalizer(brightness/contrast/saturation)適用後の画を返す。ベースラインはffmpeg由来でソース空間のため、比較の空間が食い違い、補正→測定→補正…の自己フィードバックで振動していた。`_rt_loop` 内で測定値を実適用値から逆算してソース空間に戻す処理(`k_c`/`k_s`/`b_adj` を使った逆補正)を入れて解消。**この逆補正の仕組みは今後この付近を触るときに絶対に壊さないこと**。
3. **輝度ゲイン乗算方式は使わない**: シャドウリフトの初期実装は「輝度ベースの一律ゲインをRGB全チャンネルに掛ける」方式だったが、輝度計算のB寄与が7%しかないため暗部の青みが最大3倍近く増幅されて青白く飛ぶ不具合が出た。**チャンネル別にトーンカーブを適用する方式(現行)が正解**。
4. **モード強度は実測で調整済み**: `RT_MODES = {"控えめ": (0.4, 0.10), "標準": (0.6, 0.20), "見やすさ優先": (0.9, 0.35), "極暗": (0.9, 0.35)}`(タプルは (強度倍率, 意図的暗所での残存補正下限))。彩度の自動追加は「効きすぎて違和感が出る」との評価で複数回縮小済み(現在: 基本係数3、色差追従8、上限12)。**これ以上下げる場合以外は、この方向の再調整は一旦不要という認識**。

### git履歴に関する重要な注意

セッション中に一度 `git init` で誤って別リポジトリとして作り直しかけたが、GitHub上に既存の正規履歴(v1.3〜v1.6)があることが判明し、`git update-ref refs/heads/main origin/main` で作業ツリーを保持したまま履歴だけ再接続して事なきを得た。**Codex側で作業を始める前に `git status`・`git log` で現在のHEADがGitHub `origin/main` と一致していることを確認してから着手すること**。

### PyInstaller関連の注意

- `build_exe.bat` はビルド後に `taskkill /f /im explorer.exe` を実行しExplorerを再起動する処理が入っているため、自動実行(CIやエージェントからの実行)には不向き。pyinstallerコマンドを直接叩くこと(下記)。
- `--onedir` ビルドは `dist\Lumveil` を `--clean` で作り直すため、`Lumveil_Associate.exe`(別コマンドで `--distpath dist\Lumveil` 指定)も**毎回セットで**再ビルドしないと欠落する。
- ビルド後、PyInstaller生成物(`.spec`ファイル等)と既存Anime4Kシェーダー資産に、改行コード(LF/CRLF)のみの差分がgit上で大量発生することがある。コミット前に `git diff --cached --numstat` で追加行数==削除行数のファイルを機械的に確認し、実質変更でなければ `git restore --staged` するとよい。

```bat
cd D:\tool\Lumveil
pyinstaller --onedir --windowed --name Lumveil --icon Lumveil.ico --clean --noconfirm lumveil.py
pyinstaller --onefile --windowed --name Lumveil_Associate --icon Lumveil.ico --uac-admin --distpath dist\Lumveil lumveil_associate.py
copy /Y libmpv-2.dll dist\Lumveil\
copy /Y libmpv-2.dll dist\Lumveil\_internal\
copy /Y ffmpeg.exe dist\Lumveil\
copy /Y README.md dist\Lumveil\
xcopy /E /I /Y shaders dist\Lumveil\shaders\
```

## 動作確認の作法

このプロジェクトはGUIアプリだが、以下の順で軽量に確認できる(フルGUI起動確認は最終手段):

1. `python -m py_compile lumveil.py`
2. 該当ロジックの数値部分だけ抜き出したスクラッチスクリプトで机上シミュレーション
3. `python lumveil.py` を起動し数秒後に生存確認(tkinterのUI構築ミスはこの段階でしか発覚しないため必須)
4. 実際の暗い動画での見え方はユーザー確認が必要(自動テスト不可な領域)
