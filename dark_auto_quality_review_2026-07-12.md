# Lumveil 暗所自動画質調整・機能レビュー検討書

作成日: 2026-07-12  
対象: `lumveil.py` / `shaders/lumveil_auto_contrast.glsl`（Lumveil v1.6 系）  
目的: 暗所自動画質調整の最適化方針と、Lumveil の製品特性に照らした機能・実装上の課題を整理する。

## 前提となる製品特性

Lumveil は次を特徴とするローカル動画プレイヤーである。

- 暗所自動画質調整
- 軽快に動作し、利用者が安心して使えること
- 基本的な機能が揃い、設定が充実しつつ直感的に分かること

本書では、画質だけを優先して処理を重くしたり、一般利用者に細かなパラメータ操作を求めたりする案は最適解としない。

---

## 1. 現行実装の整理

### 1.1 AUTO の処理フロー

1. AUTO 開始時、ffmpeg で動画全体から 32 点を抽出する。
2. 明るさと明暗差が一定以上のフレームを「正常」候補とし、上位 15 件の平均をベースラインにする。
3. 再生中は 0.5 秒ごとに mpv の表示フレームを取得し、64 x 36 に縮小して解析する。
4. 平均輝度・輝度の標準偏差・色差をベースラインと比較して暗所度を求める。
5. 輝度、実効コントラスト、彩度を段階的に補正する。コントラストは mpv の範囲を超える分を GLSL で拡張する。
6. 極端な暗さが 3 秒以上続く場合は、意図的な暗所として補正量を弱める。

根拠となる実装箇所:

- ベースライン解析: `lumveil.py` の `_rt_establish_baseline`
- リアルタイム判定・補正: `lumveil.py` の `_rt_loop`
- 補正値の滑らかな反映: `lumveil.py` の `_rt_blend_step`
- コントラスト拡張シェーダー: `shaders/lumveil_auto_contrast.glsl`

### 1.2 現行の良い点

- 再生中の解析では ffmpeg を起動し直さず、`player.screenshot_raw()` を使っている。継続的なプロセス起動やディスクアクセスを避けており、軽快さに配慮されている。
- ベースラインを動画全体から取るため、単純な固定しきい値より動画ごとの差に追随しやすい。
- EMA で補正値を徐々に反映し、画面が急に明滅することを抑えている。
- 明るさを最大 +3 に抑え、画面全体を白っぽくしすぎない意図がある。
- 手動調整の基準値を保存し、AUTO 停止時には元の手動値へ戻すため、利用者の調整を壊しにくい。

## 2. 暗所自動画質調整の評価

### 2.1 結論

コントラスト主導は、暗所の「情報を見やすくする」目的には不十分であり、ケースによっては逆効果になる。

推奨する最適解は、次の組合せである。

> **暗部限定のトーンマッピングを主処理にし、画面内の暗部比率を用いた場面判定、演出保護、控えめな副次補正を組み合わせる。**

これは AI による復元や重い超解像ではない。既存の mpv/GLSL パイプラインで実現可能な、軽量で予測可能な補正を優先する方針である。

### 2.2 コントラストだけでは解決できない理由

コントラストは画素値の差を広げる処理であり、黒つぶれによって失われた階調そのものを復元できない。暗部を下げやすい特性もあるため、暗い映像に強く掛けると見たい被写体がより見えなくなることがある。

現行シェーダーは pivot を 0.15 にして黒つぶれを避けようとしており、この判断自体は妥当である。しかし pivot 固定の線形コントラスト拡張では、映像ごとの暗部位置、顔や被写体の局所的な暗さ、HDR/SDR の見え方までは扱えない。

### 2.3 暗所の種類ごとの問題と対応

| 暗所の種類 | 現行方式で起こり得ること | 望ましい対応 |
|---|---|---|
| 黒つぶれ・露出不足 | コントラストを上げても情報は戻らず、黒や白の飽和を増やし得る | 暗部だけを緩やかに持ち上げ、中間調・ハイライトは保護する |
| 被写体だけが暗い逆光 | 全画面平均は十分明るく、AUTO が発動しない | 画面を粗く分割し、暗い領域の割合・偏りを見る |
| ノイズの多い夜景 | ノイズが明暗差として扱われ、補正量が不安定または過小になる | ノイズ兆候時は補正を抑え、必要時だけ軽いノイズ低減を選択可能にする |
| 霧・白い壁・淡いアニメ背景 | 明るいが低コントラストな画面を暗所と誤認する | 平均輝度が十分高い場合は、標準偏差だけで暗所判定しない |
| 意図的な暗転・ホラー演出 | 補正が演出を壊す | 急な場面転換や長時間の極端な暗部では補正を素早く抑制・停止する |
| HDR / 外部シェーダー使用時 | 出力設定・シェーダーとの組合せにより補正結果が予測しにくい | HDR・外部シェーダー時は保守モード、または AUTO 停止を明示する |

### 2.4 現行判定における具体的な懸念

現行の暗所度は、平均輝度比と標準偏差比の小さい方を採用する。

```python
raw_ratio = min(ratio_mean, ratio_std)
```

このため、平均輝度は高くても標準偏差が低い映像を暗所と判定し得る。たとえば、霧の場面、白基調の低コントラスト映像、フラットなアニメ背景が該当する。

また、極端な暗所を意図的な演出として扱う処理はあるが、3 秒後も補正量を最大 60% 残す。演出を優先するなら、既定値はより停止寄りであるべきである。

### 2.5 推奨アルゴリズム

#### 判定

64 x 36 の低解像度解析を維持し、負荷を増やさない。平均輝度・標準偏差に加え、次を使う。

- 暗部率: 輝度が低い画素の割合
- 中間調率: 極端な黒・白以外の画素の割合
- 画面分割した暗部率: 例として 4 x 3 のブロックごとの暗部率
- 直前フレームとの差: 急なシーンチェンジの検出

判定例:

- 平均輝度が低い、かつ暗部率が高い場合のみ通常の暗所候補にする。
- 平均輝度が高い低コントラスト画面は補正しない。
- 一部ブロックだけ極端に暗い場合は、全画面補正を弱める。局所補正を実装する場合も、強さは控えめにする。
- 大きな場面変化の直後は 0.5〜1 秒ほど補正更新を保留し、新しい場面へ追随する。

#### 補正

1. **主処理: 暗部限定トーンカーブ**  
   シャドウだけを持ち上げ、一定以上の中間調・ハイライトには影響をほぼ与えない GLSL 処理を使う。固定の全域コントラストではなく、滑らかな toe カーブまたは shadow lift を想定する。

2. **副処理: 小さなコントラスト補正**  
   暗部を持ち上げて眠くなった映像だけに、低い上限で適用する。現行の +300 相当を標準運用の中心にしない。

3. **副処理: 小さな彩度補正**  
   暗部を持ち上げた後に色が抜けた場合のみ、控えめに使う。

4. **保護処理**  
   ハイライト保護、補正上限、シーンチェンジ時の保留、意図的暗所の停止寄り制御を組み込む。

#### 利用者向けの見せ方

利用者に数値を判断させないため、AUTO の既定 UI は次の 4 段階でよい。

| モード | 目的 | 補正方針 |
|---|---|---|
| OFF | 原画優先 | 補正しない |
| 控えめ | 演出優先 | シャドウを少しだけ持ち上げる |
| 標準 | 推奨既定値 | 暗部を見やすくし、演出保護を強くする |
| 見やすさ優先 | 明るい視聴環境 | 暗部補正を強めるが、ハイライトは保護する |

詳細な閾値や上限は「詳細設定」に退避する。通常画面に平均輝度比・ガンマ・コントラスト値を常時出す必要はない。

### 2.6 パフォーマンスと安全性

- 再生中のフレーム解析は現行どおり低解像度・低頻度で十分である。
- AUTO 開始時の 32 点 ffmpeg 解析は、動画やストレージによってベースライン確立が遅くなり得る。まず再生中フレームで仮判定し、バックグラウンドで基準値を完成させる設計が望ましい。
- ベースライン解析中に動画を切り替えた場合、古い動画の結果を新しい動画に適用しないよう、動画パスまたは世代番号を結果と一緒に検証する。
- HDR、外部 GLSL、GPU 設定が特殊な場合は、AUTO の状態表示で「保守モード」または「この組合せでは補正を停止」と明示する。

## 3. プレイヤー機能・コードのレビュー

### 3.1 既に備わっている強み

以下は実装済みであり、「一通り揃っている」プレイヤーの基盤として十分である。

- 再生、一時停止、シーク、コマ送り、再生速度、音量、全画面、常に手前
- フォルダ連続再生、終了時動作、並び順、再開位置、最近開いたファイル
- 音声トラック・字幕トラック選択、音声/字幕遅延
- A-B リピート、ブックマーク、スクリーンショット、サムネイル付きシーク
- 手動画質調整、画質モード、GPU デコード、スケーラー、デバンド、HDR トーンマッピング、デインターレース
- Anime4K プリセットと外部 GLSL シェーダー
- 操作バーの自動省略と、設定からの表示項目カスタマイズ

### 3.2 優先課題

| 優先度 | 課題 | 理由・提案 |
|---|---|---|
| 最優先 | AUTO の品質保証 | 本製品固有の価値である。上記の判定・補正・場面保護を先に整える。 |
| 最優先 | AUTO の動画切替安全性 | 非同期ベースライン結果が別動画に混ざらないよう、動画識別子または世代番号で検証する。 |
| 高 | 失敗が見えない操作の削減 | mpv 操作・設定保存で例外を握りつぶす箇所が多い。重要な失敗は短い通知と診断ログを残す。 |
| 高 | 画質プリセットの導入 | 詳細 GPU 設定が多いため、`軽快`、`標準`、`アニメ高画質`、`暗所優先` を先に提示する。 |
| 高 | 軽快モード | 高負荷な Anime4K、AMF 補間、デバンド、AUTO を一括で安全側に戻せるようにする。 |
| 中 | 再生リストの見える化 | 現在のフォルダ内再生順・次の動画を確認し、選択できるパネルがあると実用性が上がる。 |
| 中 | チャプター操作 | 長尺動画・映画向けにチャプター一覧と前後移動を追加候補とする。 |
| 中 | 外部字幕の導線 | 外部字幕読み込み、字幕スタイル、遅延調整への導線を整理する。 |

### 3.3 UI/UX 上の考察

設定機能自体は充実している。一方で、GPU 設定はスケーラー、クロマスケール、デバンド、シグモイド、ディザリング、トーンマッピング、外部 GLSL など判断の難しい選択肢が多い。

推奨する構成:

- 最初の画面: 用途プリセットと、AUTO の4段階だけを中心に表示する。
- 詳細設定: 現在の GPU/シェーダー設定を維持する。
- GPU 入口: 独立ボタンより「設定」内の詳細タブとして見せる。既に設定ウィンドウ内へ統合されているため、入口も統一すると迷いにくい。
- ステータス: 技術値の常時表示より、`暗部を控えめに補正中`、`演出保護のため補正を抑制中`、`軽快モード` のような意味が分かる表現を優先する。

### 3.4 不要機能について

明確に削除すべき機能は現時点ではない。高度な GPU/GLSL 機能は一般利用者向けの主導線から外せばよく、画質を追求したい利用者向けの詳細設定として価値がある。

削減すべきなのは機能そのものではなく、初期画面で同時に見せる判断項目である。

## 4. 実装前に確認すべき検証項目

暗所 AUTO は動画の種類による差が大きいため、実装判断は静止画だけで行わない。以下の実動画群で、OFF と各 AUTO モードを比較する。

1. 露出不足で人物・物体が見えない夜景
2. 明るい窓を背景に人物だけ暗い逆光
3. ノイズの多いスマートフォン夜間動画
4. 意図的な暗転、ホラー、映画の夜景
5. 霧、雪、白い壁、淡いアニメ背景などの低コントラスト明所
6. SDR と HDR 素材
7. 字幕、Anime4K、デバンド、AMF 補間の有無を変えた組合せ

評価項目:

- 被写体の視認性
- 黒つぶれ・白飛び・色飽和の増減
- ノイズの目立ち方
- 意図した暗い演出を壊していないか
- 場面切替時のちらつき・追従遅延
- CPU/GPU 使用率、フレームドロップ、操作感
- AUTO を OFF に戻した際に手動値と見た目が正しく復元されるか

## 5. 推奨する進行順

1. **方針確定**: AUTO を「暗部限定トーンマッピング中心」にすること、利用者向けモードを4段階にすることを決める。
2. **最小実装**: 新しい GLSL を追加し、既存の簡易判定へ暗部率を追加する。動画切替時の世代チェックも同時に入れる。
3. **比較検証**: 上記の動画群で、現行 AUTO と最小実装を比較する。
4. **仕上げ**: 場面転換保護、HDR/外部シェーダー時の保守動作、用途プリセット、説明文を整える。

この順なら、Lumveil の強みである軽快さと安心感を保ったまま、固有機能である暗所自動画質調整の実効性を高められる。
